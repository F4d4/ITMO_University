# Информационная система управления транспортными средствами

В этом одном проекте выполнены все задания для лабораторных работ с первой по четвертую. Проект представляет собой полнофункциональную информационную систему для работы с коллекцией объектов `Vehicle`: серверная часть реализована на Java/Jakarta EE, клиентская часть - на React.

## Состав проекта

- `backend` - Java backend, собираемый в WAR-архив `project.war`.
- `frontend` - React-приложение для работы с системой через браузер.
- `test_files` - примеры JSON-файлов для проверки импорта.
- `diagram.puml`, `diagram2.puml`, `diagram3.puml` - PlantUML-диаграммы архитектуры и логики системы.
- Gradle-проект состоит из двух модулей: `backend` и `frontend`.

## Используемые технологии

- Java 17
- Jakarta EE 10: JAX-RS, CDI, EJB, WebSocket
- Hibernate ORM 6
- PostgreSQL
- Apache Commons DBCP2
- Ehcache / Hibernate second-level cache
- MinIO
- Jackson
- React 18
- React Router
- Axios
- Gradle

## Что реализовано

### Работа с коллекцией Vehicle

Реализована предметная область для транспортных средств:

- `Vehicle`
- `Coordinates`
- `VehicleType`
- `FuelType`

Для объектов реализованы проверки ограничений:

- имя транспортного средства не может быть пустым;
- координаты обязательны;
- значение `y` не превышает `621`;
- `enginePower`, `numberOfWheels`, `capacity`, `fuelConsumption` должны быть больше нуля;
- `distanceTravelled` не может быть отрицательным;
- дата создания генерируется автоматически;
- идентификаторы генерируются автоматически.

### REST API

Backend предоставляет REST API для работы с объектами `Vehicle` и `Coordinates`.

Реализованы операции:

- создание транспортного средства;
- получение объекта по ID;
- получение списка с пагинацией;
- фильтрация и сортировка списка;
- обновление объекта;
- удаление объекта;
- получение объекта с максимальной вместимостью;
- поиск по префиксу имени;
- поиск по расходу топлива больше заданного значения;
- поиск по типу транспортного средства;
- сброс пробега транспортного средства до нуля;
- получение списка координат;
- получение координат по ID.

### Работа с базой данных

Доступ к данным реализован через DAO и Hibernate.

В проекте есть:

- DAO-слой для сущностей;
- Hibernate XML-маппинги;
- подключение к PostgreSQL;
- транзакционная работа с данными;
- автоматическое обновление схемы;
- программные проверки уникальности бизнес-ограничений.

### Frontend

React-приложение предоставляет интерфейс для основных сценариев работы:

- просмотр таблицы транспортных средств;
- пагинация;
- фильтрация;
- сортировка;
- поиск объекта по ID;
- создание нового `Vehicle`;
- просмотр подробной информации;
- редактирование;
- удаление;
- выполнение специальных операций;
- импорт данных из JSON;
- просмотр истории импорта;
- скачивание файла импорта.

### Импорт данных

Реализован импорт массива объектов `Vehicle` из JSON-файла.

Особенности импорта:

- проверка корректности JSON;
- проверка, что файл содержит массив объектов;
- валидация каждого импортируемого объекта;
- импорт всех объектов в рамках одной операции;
- сохранение истории импорта;
- отображение количества успешно добавленных объектов;
- сохранение ошибки при неуспешном импорте;
- разделение прав доступа к истории импорта;
- администратор видит все операции, обычный пользователь - только свои.

### Распределенная транзакция и MinIO

Для импорта реализован координатор распределенной транзакции между базой данных и MinIO.

Логика построена по принципу двухфазного коммита:

1. Prepare-фаза: подготовка базы данных и файла для MinIO.
2. Commit-фаза: фиксация файла в MinIO и фиксация изменений в базе данных.
3. Rollback: при ошибке выполняется откат базы данных и удаление файла из MinIO, если он был загружен.

Файл импорта сохраняется в MinIO, а в истории операции хранится имя сохраненного файла.

### Кэширование

В проекте подключен Hibernate second-level cache через Ehcache.

Реализованы:

- конфигурация L2 cache;
- query cache;
- сбор статистики Hibernate;
- REST API для просмотра статистики кэша;
- очистка статистики;
- очистка всех кэшей;
- очистка кэша конкретной сущности;
- включение и отключение статистики.

### AOP и интерцепторы

В backend реализованы CDI-интерцепторы:

- `@Logged` - логирование вызовов методов, параметров, времени выполнения и ошибок;
- `@ImportLock` - пессимистическая блокировка операций импорта;
- `@CacheStatistics` - сбор и логирование статистики кэша.

### WebSocket-уведомления

Реализован WebSocket endpoint `/ws/vehicles`.

Через него frontend получает real-time уведомления о событиях:

- создание `Vehicle`;
- обновление `Vehicle`;
- удаление `Vehicle`;
- завершение операции импорта.

После получения уведомлений интерфейс автоматически обновляет список объектов или историю импорта.

## Основные API endpoints

### Vehicle

- `GET /api/vehicles` - список объектов с пагинацией, фильтрацией и сортировкой.
- `GET /api/vehicles/{id}` - получение объекта по ID.
- `POST /api/vehicles` - создание объекта.
- `PUT /api/vehicles/{id}` - обновление объекта.
- `DELETE /api/vehicles/{id}` - удаление объекта.
- `GET /api/vehicles/max-capacity` - объект с максимальным `capacity`.
- `GET /api/vehicles/by-name-prefix?prefix=...` - поиск по префиксу имени.
- `GET /api/vehicles/by-fuel-consumption?minConsumption=...` - поиск по расходу топлива.
- `GET /api/vehicles/by-type?type=...` - поиск по типу.
- `POST /api/vehicles/{id}/reset-distance` - сброс пробега.

### Coordinates

- `GET /api/coordinates` - список координат.
- `GET /api/coordinates/{id}` - координаты по ID.

### Import

- `POST /api/import/vehicles` - импорт массива `Vehicle` из JSON.
- `GET /api/import/history` - история импорта.
- `GET /api/import/download/{fileName}` - скачивание файла импорта из MinIO.

### Cache

- `GET /api/cache/statistics` - статистика кэша.
- `POST /api/cache/statistics/clear` - очистка статистики.
- `POST /api/cache/evict/all` - очистка всех кэшей.
- `POST /api/cache/evict/{entityName}` - очистка кэша сущности.
- `POST /api/cache/statistics/enable` - включение статистики.
- `POST /api/cache/statistics/disable` - отключение статистики.
