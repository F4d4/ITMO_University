# Методика тестирования

Документ описывает постановку задачи, тестовый стенд, устройство тест-плана JMeter, параметры прогонов и критерии оценки. Результаты приведены в [load-results.md](load-results.md) и [stress-results.md](stress-results.md); теоретическая база — в [theory.md](theory.md).

## 1. Постановка задачи (вариант)

Провести нагрузочное и стресс-тестирование веб-приложения средствами Apache JMeter.

| Параметр | Значение |
|---|---|
| Тестируемый URL | `http://stload.se.ifmo.ru:8080/?token=519244622&user=-1333019307&config=N` |
| Конфигурация N=1 | $3000 |
| Конфигурация N=2 | $3600 |
| Конфигурация N=3 | $5900 |
| Макс. число параллельных пользователей | 5 |
| Средняя нагрузка одного пользователя | 20 запросов в минуту |
| **Номинальная суммарная нагрузка** | 5 × 20 = **100 запр./мин ≈ 1.667 запр./с** |
| **Макс. допустимое время обработки запроса (SLA)** | **840 мс** |
| HTTP 403 | некорректные параметры запроса |
| HTTP 503 | сервер не справляется с нагрузкой |

**Задача 1 (нагрузочное):** прогнать 3 конфигурации при нагрузке 100 запр./мин и выбрать самую дешёвую, у которой время отклика укладывается в 840 мс.

**Задача 2 (стресс):** для выбранной конфигурации ступенчато повышать нагрузку, найти точку, где время отклика превышает SLA (или появляются 503), и построить график «время отклика vs нагрузка».

Производные величины: период между запросами одного пользователя = 60/20 = **3000 мс**; потолок пропускной способности при `N` потоках = `N / R` (закон Литтла, см. [theory.md](theory.md#32-закон-литтла)).

## 2. Тестовый стенд

Тестируемый сервер доступен **только из внутренней сети кафедры** — с локальной машины он недоступен. Поэтому JMeter установлен и запускается **на учебной машине `helios` (`ssh hel`)**, находящейся во внутренней сети:

| Компонент | Значение |
|---|---|
| Машина нагрузки | `helios.cs.ifmo.ru` (FreeBSD 14.4, 16 CPU, 137 ГБ ОЗУ) |
| Подключение | `ssh hel` (`se.ifmo.ru:2222`) |
| Java | `/usr/local/openjdk17` |
| Apache JMeter | 5.6.3 (`~/apache-jmeter-5.6.3`), режим non-GUI |
| Плагины для графиков | `jpgc-graphs-basic`, `jpgc-graphs-vs` (KPI vs KPI), CMDRunner |

**Почему на `hel`, а не через SSH-туннель.** Прогон JMeter локально с пробросом трафика через `ssh -L` добавил бы к каждому запросу задержку туннеля и мультиплексирования, исказив абсолютные значения времени отклика относительно порога 840 мс. Запуск на `hel` измеряет время «как видит его внутренняя сеть».

**Особенность Java на стенде.** На машине 137 ГБ ОЗУ, поэтому JVM по умолчанию пытается зарезервировать ~¼ ОЗУ под heap (~30 ГБ), что превышает пользовательский лимит `ulimit -d` (32 ГБ) и приводит к падению. Поэтому heap **всегда** ограничивается: `export JVM_ARGS="-Xms512m -Xmx2g"`.

## 3. Тест-план JMeter (`load-test.jmx`)

Единый параметризуемый план для всех прогонов (и нагрузочного, и стресс). Дерево элементов:

```
Test Plan  (User Defined Variables: TARGET_HOST, TARGET_PORT, TOKEN, USER,
│           CONFIG=${__P(config,2)}, THROUGHPUT=${__P(throughput,100)},
│           THREADS=${__P(threads,5)}, DURATION=${__P(duration,180)}, RAMPUP=${__P(rampup,10)})
├── HTTP Request Defaults   (stload.se.ifmo.ru:8080, GET /, connect 5s / response 30s)
└── Thread Group "Users"    (threads=${THREADS}, ramp-up=${RAMPUP}, scheduler, duration=${DURATION})
    ├── HTTP Request "GET config"   (token=${TOKEN}, user=${USER}, config=${CONFIG})
    ├── Constant Throughput Timer   (target=${THROUGHPUT}/min, "all active threads in current thread group")
    ├── Response Assertion           (Response Code = 200)
    └── Simple Data Writer           (JTL/CSV: timeStamp, elapsed, responseCode, success, allThreads, Latency, Connect)
```

**Параметризация.** Функция `${__P(имя,умолчание)}` читает property, заданное флагом `-Jимя=...`. Один и тот же `load-test.jmx` запускается для разных конфигураций и уровней нагрузки без правки XML:
`-Jconfig` (1/2/3), `-Jthroughput` (целевой req/min), `-Jthreads`, `-Jduration`, `-Jrampup`.

**Constant Throughput Timer.** Цель — `THROUGHPUT` сэмплов/мин, режим **«all active threads in current thread group»**: таймер делит целевую интенсивность между активными потоками. При 5 потоках и цели 100 это даёт 100/5 = 20 запр./мин на поток — ровно условие варианта. Таймер только ограничивает интенсивность сверху (добавляет паузы), не может превысить потолок `N/R`.

**Response Assertion на код 200.** Ответы 403/503 помечаются `success=false`, попадают в долю ошибок и **исключаются из расчёта времени отклика** (быстрый отказ 503 иначе занизил бы среднее).

## 4. Параметры прогонов

**Нагрузочный тест** ([run-load.sh](../run-load.sh)) — по очереди 3 конфигурации при штатной нагрузке:

| Параметр | Значение |
|---|---|
| threads | 5 (максимум по варианту) |
| throughput | 100 запр./мин |
| ramp-up | 10 с |
| duration | 180 с (≈ 300 запросов на конфигурацию) |
| пауза между конфигурациями | 15 с |

**Стресс-тест** ([run-stress.sh](../run-stress.sh)) — для выбранной cfg2 серия отдельных прогонов с растущей нагрузкой. Поскольку при 5 потоках потолок ≈ `5/0.72 ≈ 419` запр./мин, для нагрузки выше номинала число потоков увеличивается:

| Уровень (target, запр./мин) | 100 | 200 | 300 | 400 | 500 | 600 | 800 | 1000 | 1200 |
|---|---|---|---|---|---|---|---|---|---|
| Потоков | 5 | 8 | 12 | 16 | 20 | 25 | 35 | 45 | 55 |

Каждый уровень — 75 с, пауза 15 с. **Серия плоских прогонов** выбрана вместо Stepping Thread Group: каждый уровень даёт «чистую» статистику (не размазанную по растущей нагрузке внутри одного прогона), один файл = одна точка графика, без внешних зависимостей.

## 5. Команды запуска (на `hel`)

```bash
export JAVA_HOME=/usr/local/openjdk17
export JVM_ARGS="-Xms512m -Xmx2g"           # обязательно: ограничить heap
JM=~/apache-jmeter-5.6.3/bin/jmeter

# нагрузочный прогон одной конфигурации + HTML-дашборд
"$JM" -n -t load-test.jmx -l results/load_cfg2.jtl -e -o results/report_cfg2 \
      -Jconfig=2 -Jthroughput=100 -Jthreads=5 -Jduration=180 -Jrampup=10
```

Флаги: `-n` non-GUI, `-t` план, `-l` JTL-результаты, `-e -o` HTML-дашборд после прогона, `-J` property.

## 6. Метрики и критерии

**Обработка результатов без Python** — скриптом [metrics.sh](../metrics.sh) (awk/sort/sed): по каждому JTL считаются avg, P90, P95, P99, max времени отклика и доля ошибок.

**Окно прогрева.** Первый запрос каждого прогона к паре `(user, config)` обслуживается с холодным стартом сервера (~2.6 с) и одинаково завышает `max` у всех конфигураций. Поэтому из расчёта метрик исключаются первые **15 с** каждого прогона (`WARM=15`). Тайминги считаются только по успешным (HTTP 200) сэмплам.

**Метрики времени отклика.** По заданию приводятся **все три**: среднее (avg), 95-й перцентиль (P95) и максимум (max). На данном сервере время отклика очень детерминировано, поэтому метрики согласованы.

**Критерий выбора конфигурации:** конфигурация удовлетворяет SLA, если время отклика ≤ 840 мс и доля ошибок = 0 при нагрузке 100 запр./мин. Среди удовлетворяющих выбирается минимальная по стоимости.

**Критерий «перестаёт удовлетворять» (стресс):** наименьшая нагрузка, при которой время отклика впервые превышает 840 мс **либо** появляются ответы 503.

## 7. Построение графиков (средствами JMeter)

Графики строятся **самим JMeter**, без сторонних средств визуализации:

- **HTML-дашборды** (`-e -o`) — для каждого прогона: Response Times Over Time, перцентили, Active Threads Over Time, доля кодов ответа.
- **CMDRunner + плагин `jpgc-graphs-vs`** — итоговый график стресс-теста в headless-режиме:
  ```bash
  java -jar lib/cmdrunner-2.3.jar --tool Reporter \
       --generate-png stress_TimesVsThreads.png \
       --input-jtl results/stress/stress_merged.jtl --plugin-type TimesVsThreads
  ```
  `TimesVsThreads` — время отклика (y) от числа активных пользователей-потоков (x, мера нагрузки); `ThroughputVsThreads` — фактическая пропускная способность от нагрузки (показывает насыщение).
