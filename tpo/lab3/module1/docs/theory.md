# 1. Функциональное тестирование

## 1.1. Определение

**Функциональное тестирование (functional testing)** — вид тестирования программного обеспечения, при котором система проверяется с точки зрения **соответствия её поведения функциональным требованиям**: «при таких-то входных данных и таком-то предусловии система должна выдать такой-то результат».

В отличие от **нефункционального** тестирования (производительность, нагрузочное, безопасность, юзабилити) функциональное отвечает на вопрос **«что делает система?»**, а не «насколько хорошо/быстро/безопасно».

Стандарт ISO/IEC 25010 относит функциональную пригодность к одной из восьми характеристик качества ПО и выделяет в ней три подхарактеристики:

| Подхарактеристика | Что проверяется |
|---|---|
| Functional completeness | Полнота функций — реализованы ли все заявленные возможности |
| Functional correctness | Корректность — выдаёт ли система верный результат для каждого входа |
| Functional appropriateness | Уместность — действительно ли реализованные функции решают задачу пользователя |

## 1.2. Базовые понятия

| Термин | Значение |
|---|---|
| **Test case** (тест-кейс) | Описание одной проверки: предусловие, шаги, ожидаемый результат |
| **Test suite** (тест-сюита) | Группа связанных тест-кейсов, запускаемых вместе |
| **Test plan** | Документ верхнего уровня: что, как, кем и когда тестируется |
| **Fixture** | Фиксированное окружение для теста (данные, состояние БД, поднятые сервисы) |
| **Assertion** | Утверждение о состоянии системы; провал ассерта = провал теста |
| **Oracle** | Источник истины — то, с чем сравнивается фактический результат |
| **SUT** (System Under Test) | Тестируемая система или её часть |
| **Defect / bug** | Несоответствие между фактическим и ожидаемым поведением |
| **Regression** | Возврат старого дефекта или появление нового из-за изменений |
| **Coverage** | Доля требований/кода/сценариев, покрытая тестами |
| **Traceability matrix** | Матрица соответствия «требование ↔ тест-кейс ↔ дефект» |

## 1.3. Подходы по доступу к внутренностям

| Подход | Что известно тестировщику | Применение |
|---|---|---|
| **Чёрный ящик** (black-box) | Только спецификация, исходников не видно | Системные, приёмочные, UI-тесты — например, наш проект |
| **Белый ящик** (white-box) | Полный доступ к исходному коду | Модульные тесты, code coverage |
| **Серый ящик** (gray-box) | Частичный — известна архитектура, БД, API | Интеграционные тесты |

**Функциональное тестирование** — обычно чёрный или серый ящик. Selenium-тесты в нашем проекте — классический чёрный ящик: проверяем поведение Stack Overflow, не имея доступа к их коду.

## 1.4. Уровни функционального тестирования

Стандартная V-модель связывает уровни тестирования с уровнями разработки:

```
Требования     ─────────►  Приёмочные тесты   (acceptance)
   │                              ▲
Системный      ─────────►  Системные тесты    (system)
дизайн                            ▲
   │                              ▲
Архитектура    ─────────►  Интеграционные     (integration)
   │                              ▲
Детальный      ─────────►  Модульные тесты    (unit)
дизайн                            ▲
   │                              ▲
   └─────► Реализация ────────────┘
```

| Уровень | Что тестируется | Кто пишет | Инструменты |
|---|---|---|---|
| **Модульный** (unit) | Отдельные функции/классы в изоляции, моки внешних зависимостей | Разработчики | JUnit, pytest, Jest |
| **Интеграционный** (integration) | Взаимодействие нескольких модулей | Разработчики/QA | Testcontainers, Spring Test |
| **Системный** (system) | Вся собранная система целиком | QA | Selenium, Cypress, Playwright |
| **Приёмочный** (acceptance) | Соответствие бизнес-сценариям; подтверждение, что продукт готов к выпуску | Заказчик/QA | Cucumber, Robot Framework, ручное тестирование |

Наш Selenium-проект — **системный + приёмочный** уровень: он проверяет поведение реального сайта Stack Overflow с позиции конечного пользователя.

## 1.5. Виды функционального тестирования

| Вид | Описание |
|---|---|
| **Smoke** | Поверхностная проверка — «дымит или не дымит». Несколько критичных сценариев, цель — отсеять явно нерабочую сборку до прогона полного регресса |
| **Sanity** | Проверка узкой функциональности после небольшого изменения — «починили ли именно то, что должны были» |
| **Regression** | Полный (или большой) набор тестов, гоняемый после каждого релиза/PR, чтобы убедиться, что ничего старого не сломалось |
| **Re-test (confirmation)** | Перепрогон конкретного бывшего падения после фикса |
| **Exploratory** | Ручное тестирование без заранее написанного скрипта — поиск дефектов через свободное исследование |
| **Usability/UI** | Проверка UI и удобства использования (часто пересекается с нефункциональным) |
| **Acceptance** | UAT — пользовательское приёмочное; BAT — бизнес-приёмочное |

## 1.6. Use-case-based тестирование

**Use-case** (прецедент использования) — описание сценария взаимодействия актора с системой для достижения цели. Структура классического use-case:

| Поле | Пример из нашей лабы (UC-06) |
|---|---|
| **ID** | UC-06 |
| **Название** | Попытка задать вопрос без входа |
| **Актор** | Гость |
| **Предусловие** | Пользователь не авторизован |
| **Основной сценарий** | Открыть `/questions` → нажать `Ask Question` |
| **Альтернативные сценарии** | (нет — основной перекрывает все ветки) |
| **Постусловие / результат** | Сайт показывает страницу входа |

Каждому use-case в нашем проекте соответствует один Java-метод теста (`anonymousVisitorIsRedirectedToLoginWhenAskingQuestion`) и один блок в `.side`-файле Selenium IDE (`uc-06-anonymous-ask`). Это и есть **тестовое покрытие на основании прецедентов** — требование лабы.

Use-case-based тестирование особенно подходит для UI и системного уровня, потому что:
- Каждый use-case = одна end-to-end история пользователя
- Легко сопоставить с требованиями (traceability)
- Понятно заказчику без знания кода

## 1.7. Техники тест-дизайна

Когда use-case задан, нужно решить, **какие конкретно входы и шаги** проверять. Используются формальные техники:

### 1.7.1. Эквивалентные классы (equivalence partitioning)

Делим всё пространство входов на классы, внутри которых система ведёт себя одинаково. Берём по одному представителю от класса.

Пример (поле «возраст», 0..150):
- Класс 1: < 0 — невалидный
- Класс 2: 0..17 — несовершеннолетний
- Класс 3: 18..150 — совершеннолетний
- Класс 4: > 150 — невалидный

Достаточно проверить по одному значению из каждого, например: `-5, 10, 30, 200`.

### 1.7.2. Граничные значения (boundary value analysis)

Большинство ошибок возникает на **границах** интервалов (off-by-one). Берём значения **на границе** и **по обе стороны от неё**.

Для интервала `0..17` граничные точки: `-1, 0, 1, 16, 17, 18`.

### 1.7.3. Таблицы решений (decision tables)

Для функций с несколькими условиями строится таблица: строки — комбинации входов, столбцы — действия.

| Авторизован | Подписка активна | Возраст 18+ | Доступ к контенту? |
|---|---|---|---|
| Нет | — | — | Нет |
| Да | Нет | — | Превью |
| Да | Да | Нет | Без 18+ контента |
| Да | Да | Да | Полный |

### 1.7.4. Диаграммы причин и следствий (cause-effect graph)

Графовое представление таблицы решений: узлы — условия (причины) и действия (следствия), рёбра — связи. Помогает выявить избыточные/противоречивые правила.

### 1.7.5. Pairwise (попарное) тестирование

Если на вход подаётся `n` параметров с `k_i` значениями каждый, полный перебор — произведение `k_1 * k_2 * ... * k_n` — комбинаторный взрыв. Эмпирически большинство дефектов вызывается взаимодействием **только двух** параметров. Pairwise генерирует минимальный набор тестов, в котором **каждая пара значений встречается хотя бы один раз**.

Инструменты: PICT (Microsoft), allpairs.

### 1.7.6. State transition testing

Если у системы есть состояния и переходы (например, форма заказа: `пустая → заполненная → отправленная → подтверждённая`), тесты строятся по диаграмме состояний — каждое валидное и невалидное ребро.

### 1.7.7. Сценарное (scenario) тестирование

Сценарии = use-case'ы или user story. Каждый — последовательность шагов. **Это и есть подход нашей лабы.**

## 1.8. Способы организации функционального тестирования

### 1.8.1. Ручное vs автоматическое

| Критерий | Ручное | Автоматическое |
|---|---|---|
| Скорость одного запуска | Медленно | Быстро |
| Стоимость поддержки | Низкая | Высокая (тесты тоже надо чинить) |
| Чувствительность к UX-проблемам | Высокая (человек видит «странность») | Низкая (видит только то, что проверяет) |
| Подходит для | Exploratory, usability, новые фичи | Regression, smoke, длинные сценарии |
| Воспроизводимость | Низкая (человеческий фактор) | Высокая |

На практике используется **гибрид**: критичные сценарии автоматизируются, новые/исследовательские — гоняются вручную.

### 1.8.2. CI/CD интеграция

Автоматические тесты обычно интегрируются в pipeline:

```
push → build → unit → integration → e2e → deploy_staging → smoke → deploy_prod
```

На каждом этапе — свой уровень тестов. Чем «правее», тем медленнее тесты, но шире покрытие. Принцип **Test Pyramid** (Mike Cohn): много быстрых модульных, меньше интеграционных, минимум медленных e2e.

### 1.8.3. Подходы к разработке тестов

| Подход | Идея |
|---|---|
| **Test-First / TDD** | Сначала пишется падающий тест, потом код, проходящий его |
| **BDD** (Behavior-Driven Development) | Тесты в формате Given-When-Then на естественном языке (Cucumber, SpecFlow) |
| **ATDD** (Acceptance Test Driven) | Приёмочные тесты пишутся вместе с заказчиком до разработки |
| **Test-Last** | Классика — тесты пишутся после кода |

### 1.8.4. Артефакты процесса

| Артефакт | Содержит |
|---|---|
| Test Strategy | Высокоуровневые принципы тестирования в организации |
| Test Plan | Объём, ресурсы, расписание для конкретного релиза/проекта |
| Test Design | Какие техники применяются, какие наборы данных |
| Test Cases | Конкретные шаги и ожидания |
| Test Data | Подготовленные входные наборы |
| Test Report | Результаты прогона: passed/failed/skipped, дефекты |
| Traceability matrix | Связи требование ↔ тест ↔ дефект |

В нашем проекте:
- Test Plan — раздел «Use-case'ы» в [`test-coverage.md`](test-coverage.md)
- Test Cases — Java-методы и `.side`-сценарии
- Test Report — Allure (`mvn allure:serve`)
- Traceability — таблица «Тест → UC» в `test-coverage.md`

## 1.9. Решаемые задачи

Главные задачи функционального тестирования:

1. **Подтверждение соответствия требованиям** — система делает то, что обещано
2. **Раннее обнаружение дефектов** — пока цена исправления невелика
3. **Защита от регрессий** — изменения не ломают существующее поведение
4. **Снижение бизнес-рисков** — критичные сценарии заведомо работают
5. **Документирование поведения** — тесты как живая спецификация (особенно BDD)
6. **База для рефакторинга** — без тестов изменения становятся опасными
7. **Подготовка к релизу** — формальный gate «можем ли деплоить»

## 1.10. Метрики и оценка качества

| Метрика | Что показывает |
|---|---|
| **Requirements coverage** | % требований, покрытых хотя бы одним тестом |
| **Code coverage** (line/branch/condition) | % кода, исполненного тестами (только для white/grey-box) |
| **Defect density** | Дефектов на 1000 строк кода / на use-case |
| **Defect detection percentage** | Дефекты найдены тестами / (тестами + найденные после релиза) |
| **Test pass rate** | passed / total в последнем прогоне |
| **MTTR / MTBF** | Среднее время восстановления / между отказами |

Code coverage — **необходимое, но не достаточное** условие качества: 100% покрытия ≠ отсутствие багов (тесты могут исполнять код, но не проверять корректность).

# 2. Система Selenium

## 2.1. Что такое Selenium

**Selenium** — open-source семейство инструментов для **автоматизации браузеров**. Позволяет программно открывать страницы, кликать, вводить текст, читать DOM, делать скриншоты, выполнять JavaScript и проверять поведение веб-приложений как реальный пользователь.

Создан Jason Huggins в 2004 году в ThoughtWorks. С тех пор пережил несколько архитектурных поколений; современный стандарт — **WebDriver**, утверждённый W3C в 2018 году.

## 2.2. Семейство Selenium

| Компонент | Назначение | Состояние |
|---|---|---|
| **Selenium Core** | JavaScript-библиотека, исполнявшаяся внутри страницы (исторический) | Устарел |
| **Selenium RC** (Remote Control) | Прокси-сервер, инжектировавший Selenium Core в страницу | Устарел, удалён |
| **Selenium WebDriver** | Прямое управление браузером через нативный драйвер | **Основной**, W3C-стандарт |
| **Selenium IDE** | Браузерное расширение для записи и воспроизведения сценариев | Активно для smoke/прототипов |
| **Selenium Grid** | Распределённый запуск тестов на пуле машин/браузеров | Активно |

В нашем проекте используются **WebDriver** (Java-тесты, `pom.xml` → `selenium-java 4.19.1`) и **IDE** (файл `selenium-ide/stackoverflow-public-scenarios.side`).

## 2.3. Архитектура WebDriver

Архитектура — клиент-серверная, работает поверх HTTP. До 2018 использовался JSON Wire Protocol; теперь — стандарт **W3C WebDriver**.

```
┌─────────────────────┐
│  Test code (Java)   │   <- наш StackOverflowFunctionalTest
│  + Selenium API     │
└──────────┬──────────┘
           │ Java-методы (driver.findElement, click, ...)
┌──────────▼──────────┐
│  Language bindings  │   <- selenium-java 4.19.1
│  (преобразуют API   │
│   в HTTP-запросы)   │
└──────────┬──────────┘
           │ HTTP + JSON (W3C WebDriver Protocol)
┌──────────▼──────────┐
│  Browser driver     │   <- chromedriver, geckodriver
│  (отдельный exe)    │
└──────────┬──────────┘
           │ Native browser API / DevTools
┌──────────▼──────────┐
│      Browser        │   <- Chrome, Firefox, ...
└─────────────────────┘
```

| Слой | Что делает |
|---|---|
| **Client bindings** | Языковые обвязки (Java, Python, C#, Ruby, JS) — переводят вызовы API в HTTP-запросы по стандарту W3C |
| **Browser driver** | Отдельный исполняемый файл, специфичный для браузера: `chromedriver`, `geckodriver` (Firefox), `msedgedriver`, `safaridriver`. Принимает HTTP-команды и транслирует их в нативные действия браузера |
| **Browser** | Реальный браузер. WebDriver не имитирует браузер — он управляет настоящим |

С Selenium 4 драйверы автоматически скачиваются через **Selenium Manager** (встроен в `selenium-java`) — отдельная установка `chromedriver` больше не нужна.

## 2.4. Selenium Grid (кратко)

Grid — это распределённая версия WebDriver:

```
        ┌────────┐
        │  HUB   │  <- единая точка входа
        └───┬────┘
   ┌────────┼─────────┐
┌──▼──┐  ┌──▼──┐  ┌──▼──┐
│Node1│  │Node2│  │Node3│  <- разные ОС/браузеры
└─────┘  └─────┘  └─────┘
```

Тесты запрашивают сессию у Hub'а с указанием нужных capabilities (`browser=chrome`, `os=linux`), Hub находит подходящий Node и проксирует команды. Используется для параллельного прогона и кросс-браузерных тестов в CI.

## 2.5. Selenium IDE

Расширение для Chrome и Firefox с UI:
- **Запись (record)**: запускаешь, кликаешь по сайту вручную — IDE сохраняет шаги
- **Воспроизведение (play)**: прогоняет записанный сценарий
- **Экспорт**: преобразует в код (Java/JUnit, Python/pytest и т.д.)
- **Format**: проект сохраняется в `.side`-файле — это JSON со списком команд

Команды IDE:

| Категория | Примеры |
|---|---|
| Навигация | `open`, `back`, `forward`, `refresh`, `close` |
| Действия | `click`, `type`, `sendKeys`, `select`, `mouseDown`, `dragAndDrop` |
| Ассерты | `assertText`, `assertElementPresent`, `assertTitle`, `verifyValue` |
| Ожидания | `waitForElementPresent`, `waitForText`, `pause` |
| Управление | `if`, `else`, `while`, `do/repeat if`, `times` |
| Хранение | `store`, `storeText`, `storeXpathCount`, `executeScript` |

Файл `selenium-ide/stackoverflow-public-scenarios.side` нашего проекта содержит шесть тестов с командами `open`, `waitForElementPresent`, `assertElementPresent`, `type`, `sendKeys`, `click`.

IDE удобна для прототипирования и smoke-тестов, но для серьёзной автоматизации используют WebDriver: больше контроля, версионирование как обычный код, интеграция с CI и фреймворками тестирования.

## 2.6. Ключевые объекты WebDriver API

### 2.6.1. `WebDriver`

Главный интерфейс — представляет сессию с браузером.

```java
WebDriver driver = new ChromeDriver(options);
driver.get("https://stackoverflow.com/questions");  // навигация
driver.getTitle();                                  // <title>
driver.getCurrentUrl();
driver.getPageSource();
driver.manage().window().maximize();
driver.manage().timeouts().pageLoadTimeout(Duration.ofSeconds(60));
driver.quit();                                      // закрыть все окна и драйвер
driver.close();                                     // закрыть текущее окно
```

### 2.6.2. `WebElement`

Один элемент DOM. Возвращается `findElement` / `findElements`.

```java
WebElement search = driver.findElement(By.xpath("//input[@name='q']"));
search.click();
search.sendKeys("selenium webdriver");
search.clear();
search.submit();
search.getText();
search.getAttribute("href");
search.getCssValue("color");
search.isDisplayed();   // видим ли пользователю
search.isEnabled();
search.isSelected();    // для checkbox/radio/option
```

### 2.6.3. `By` — стратегия поиска

```java
By.id("username")
By.name("q")
By.className("post-tag")
By.tagName("a")
By.linkText("Ask Question")              // только для <a> по точному тексту
By.partialLinkText("Ask")
By.cssSelector("div.s-post-summary > h3 a")
```

### 2.6.4. `WebDriverWait` и `ExpectedConditions`

Явные ожидания — главный инструмент против flakiness в современных SPA, где элементы появляются асинхронно.

```java
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(20));
WebElement btn = wait.until(ExpectedConditions.elementToBeClickable(By.xpath(...)));
wait.until(ExpectedConditions.urlContains("/questions/"));
wait.until(ExpectedConditions.titleIs("Newest Questions - Stack Overflow"));
wait.until(d -> ((JavascriptExecutor)d).executeScript("return document.readyState").equals("complete"));
```

В нашем проекте `wait` инициализируется в `setUp()` и используется по всему тесту:
```java
wait = new WebDriverWait(driver, Duration.ofSeconds(TIMEOUT_SECONDS));
```

### 2.6.5. `Actions` — низкоуровневые жесты

Эмуляция мыши и клавиатуры — наведение, drag-and-drop, нажатия с модификаторами.

```java
new Actions(driver)
    .moveToElement(menu)
    .pause(Duration.ofMillis(200))
    .click(submenu)
    .perform();

new Actions(driver).keyDown(Keys.CONTROL).click(link).keyUp(Keys.CONTROL).perform();
```

### 2.6.6. `JavascriptExecutor`

Выполнение JS в контексте страницы. Используется когда обычный API не справляется (overlay перехватывает клики, нужно прокрутить, прочитать `window.something`).

```java
((JavascriptExecutor) driver).executeScript("arguments[0].click();", element);
((JavascriptExecutor) driver).executeScript("arguments[0].scrollIntoView(true);", element);
Object readyState = ((JavascriptExecutor) driver).executeScript("return document.readyState");
```

В нашем проекте `JavascriptExecutor` применяется в двух местах:
- `clickAskQuestion()` — fallback-клик через JS, если обычный перехвачен
- `waitForDocumentReady()` — опрос `document.readyState`

## 2.7. Типы ожиданий

Главный источник нестабильности (flakiness) Selenium-тестов — гонки между скриптом и асинхронным DOM. Selenium предоставляет три механизма ожидания:

### 2.7.1. Implicit wait (неявное)

```java
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
```

Все последующие `findElement/findElements` будут ждать до 10 секунд появления элемента. **Глобальная настройка**, плохо комбинируется с explicit wait — может приводить к двойным таймаутам.

В нашем проекте implicit wait **отключён**:
```java
driver.manage().timeouts().implicitlyWait(Duration.ZERO);
```
— чтобы поведение было полностью под контролем `WebDriverWait`.

### 2.7.2. Explicit wait (явное)

`WebDriverWait` + `ExpectedConditions` — ждём конкретное условие на конкретном вызове. Гибкий, рекомендуемый подход. Это то, что используется в нашем проекте повсеместно.

### 2.7.3. Fluent wait

Подвид explicit с настраиваемым интервалом polling и игнорированием конкретных исключений:

```java
Wait<WebDriver> wait = new FluentWait<>(driver)
    .withTimeout(Duration.ofSeconds(30))
    .pollingEvery(Duration.ofMillis(500))
    .ignoring(NoSuchElementException.class);
```

### 2.7.4. Антипаттерн: `Thread.sleep()`

```java
Thread.sleep(5000);  // плохо
```

Жёсткая пауза — медленно (всегда ждёт максимум) и ненадёжно (может не хватить). Допустимо только при отладке.

## 2.8. Способы доступа к элементам страницы

Selenium предлагает 8 встроенных стратегий локации (`By.*`). Каждая имеет свою область применения, скорость и устойчивость к изменениям UI.

### 2.8.1. Сводная таблица

| Стратегия | Java | Что ищет | Скорость | Устойчивость |
|---|---|---|---|---|
| **ID** | `By.id("login")` | Элемент с `id="login"` | Самая высокая | Высокая (если ID стабилен) |
| **Name** | `By.name("q")` | Элемент с `name="q"` | Высокая | Зависит от формы |
| **Class name** | `By.className("btn")` | Один CSS-класс | Высокая | Низкая (классы часто меняют) |
| **Tag name** | `By.tagName("a")` | Все теги указанного типа | Высокая | Очень низкая |
| **Link text** | `By.linkText("Ask Question")` | `<a>` с точным текстом | Средняя | Средняя (i18n ломает) |
| **Partial link text** | `By.partialLinkText("Ask")` | `<a>` с подстрокой в тексте | Средняя | Средняя |
| **CSS selector** | `By.cssSelector("div.q > a")` | По CSS-селектору | Высокая | Зависит от селектора |
| **XPath** | `By.xpath("//a[contains(@href,'/ask')]")` | По XPath-выражению | Средняя | Зависит от выражения |

### 2.8.2. Когда что выбирать

| Ситуация | Рекомендуемая стратегия |
|---|---|
| У элемента есть стабильный ID | `By.id` (но в нашей лабе ID запрещены) |
| Поле формы с уникальным `name` | `By.name` |
| Сложная структура / поиск по тексту / поиск «снизу вверх» (родители) | `By.xpath` |
| Простой селектор по классу/потомкам | `By.cssSelector` |
| Ссылка с уникальным текстом, не зависящим от языка | `By.linkText` |

CSS быстрее XPath, но у CSS нет аналогов многим возможностям XPath: оси `parent::`, `ancestor::`, `preceding::`, текстовые предикаты `contains(text(), '...')`, функции `normalize-space()`, `translate()`.

### 2.8.3. Принципы хороших локаторов

1. **Уникальность** — должен находить ровно один нужный элемент
2. **Стабильность** — не ломаться при косметических изменениях UI
3. **Читаемость** — другой инженер должен понять, что ищется
4. **Семантика** — опираться на смысл (роль, текст, href), а не случайные классы (`css-1abc23`)
5. **Минимальная привязка к структуре** — не описывать весь путь от `<html>`

Плохо: `/html/body/div[3]/div[2]/div/form/input[1]` — сломается при любой перерисовке.
Хорошо: `//form[contains(@action,'/search')]//input[@name='q']` — описывает смысл.

В нашем проекте используется второй подход, и в `StackOverflowLocators` каждый XPath опирается на:
- `@action`, `@href`, `@name` — атрибуты с семантикой
- `class contains 's-post-summary'` — стабильные имена дизайн-системы Stack Overflow
- Видимый текст: `normalize-space()='Active'`

### 2.8.4. Page Object Pattern (POM)

Промышленный паттерн: для каждой страницы создаётся класс, инкапсулирующий локаторы и действия.

```java
class QuestionsPage {
    private final WebDriver driver;
    private final By searchInput = By.xpath("//input[@name='q']");
    private final By askButton   = By.xpath("//a[contains(@href,'/ask')]");

    QuestionsPage(WebDriver driver) { this.driver = driver; }

    SearchResultsPage search(String query) {
        driver.findElement(searchInput).sendKeys(query + Keys.ENTER);
        return new SearchResultsPage(driver);
    }
    LoginPage clickAsk() {
        driver.findElement(askButton).click();
        return new LoginPage(driver);
    }
}
```

Преимущества:
- Локаторы в одном месте — изменился сайт, правим один файл
- Тесты читаются как сценарии: `questions.search("selenium").assertHasResults()`
- Легче поддерживать большой проект

В нашем проекте паттерн упрощён: локаторы вынесены в `StackOverflowLocators`, но действия остались в тестовом классе. Для шести коротких тестов это адекватно; для большой системы стоило бы построить полноценный POM.

## 2.9. Принципы написания сценариев

### 2.9.1. Структура одного теста — Arrange / Act / Assert

```
Arrange  — подготовка окружения и данных          (driver, login, ...)
Act      — действие, которое тестируется          (click, sendKeys)
Assert   — проверка ожидаемого результата         (assertTrue, ...)
```

Пример из нашего проекта (UC-04):

```java
// Arrange
open("/tags");
waitForPresent(StackOverflowLocators.PAGE_HEADING);
waitForBodyText("Tags");

// Act
WebElement pythonTag = wait.until(elementToBeClickable(PYTHON_TAG_LINK));
pythonTag.click();

// Assert
wait.until(d -> d.getCurrentUrl().contains("python"));
assertTrue(driver.getCurrentUrl().contains("python"));
```

### 2.9.2. Изоляция тестов

В идеале — каждый тест **независим**: можно запустить любой в любом порядке. Это требует либо чистого окружения для каждого теста (новый драйвер, чистая БД), либо аккуратного teardown.

В нашем проекте тесты используют **общий** `WebDriver` (`@TestInstance.PER_CLASS` + `@Order`). Это компромисс — экономит время на старт браузера и снижает шанс anti-bot верификации, но требует, чтобы каждый тест начинался с `open("/questions")` и не оставлял состояние, ломающее следующие.

### 2.9.3. Принципы

| Принцип | Что значит |
|---|---|
| **DRY** (Don't Repeat Yourself) | Повторяющиеся шаги — в helper-методы (`open`, `waitForVisible`, `acceptCookiesIfShown`) |
| **Self-checking** | Тест сам проверяет результат через assert, не требует визуального осмотра |
| **Atomic** | Один тест — одна логическая проверка |
| **Repeatable** | Прогон 100 раз даёт 100 раз одинаковый результат |
| **Fast** | Чем быстрее, тем чаще запускают |
| **Меaningful failure** | Падение должно объяснять причину; в нашем проекте `pageSnapshot()` приклеивается к AssertionError |

### 2.9.4. Defensive coding в Selenium

Реальные сайты ведут себя плохо: всплывают cookie-баннеры, появляется reCAPTCHA, элементы перерисовываются. Хорошие тесты предусматривают:

| Проблема | Защита в нашем проекте |
|---|---|
| Cookie-баннер | `acceptCookiesIfShown()` — закрывается при каждом `open()` |
| Cloudflare/anti-bot | `waitUntilInterfaceIsAvailable()` ждёт до 180с |
| Click intercepted (overlay) | JS-fallback в `clickAskQuestion()` |
| Нестабильный редирект | Двойная проверка: URL **или** появление сообщения (UC-06) |
| Поиск не сработал | Fallback через прямой URL `/search?q=…` (UC-02) |

# 3. Язык XPath

## 3.1. Что такое XPath

**XPath** (XML Path Language) — язык запросов для **навигации по узлам XML/HTML-документа**. Стандартизован W3C:
- **XPath 1.0** — 1999, поддерживается в Selenium и большинстве браузеров
- **XPath 2.0** — 2007, базируется на XQuery, есть последовательности и сильная типизация
- **XPath 3.0** / 3.1 — 2014/2017, добавлены map'ы, массивы, лямбды

**Selenium WebDriver работает с XPath 1.0** — все примеры ниже именно про эту версию (это критично, т.к. многие удобные функции, например `lower-case()` и регэкспы, появились только в 2.0).

XPath даёт доступ к чему угодно в DOM по его смыслу (текст, атрибуты, положение, отношение к другим узлам), что делает его незаменимым для автоматизации тестов веб-приложений.

## 3.2. Модель данных

XPath видит документ как **дерево узлов** — концептуально это то же самое **DOM-дерево**, по которому работает JavaScript в браузере. В Selenium XPath-выражение в итоге исполняется через браузерный `document.evaluate(...)` и обходит реальный DOM текущей страницы.

Формально же XPath имеет собственную абстрактную модель данных (в 1.0 — XPath Data Model, в 2.0+ — XDM, XQuery and XPath Data Model). Она почти полностью совпадает с DOM, но не привязана к нему: одно и то же XPath-выражение можно применить и к DOM в браузере, и к XML-файлу из библиотеки парсера, и к XSLT-преобразованию.

**Типы узлов в модели XPath 1.0:**

| Тип | Что это | Соответствие в DOM |
|---|---|---|
| **Element** | HTML-тег (`<div>`, `<a>`, ...) | `Element` |
| **Attribute** | Атрибут тега (`href`, `class`, ...) | `Attr` |
| **Text** | Текстовое содержимое | `Text` (`CDATASection` сливается с Text) |
| **Comment** | `<!-- ... -->` | `Comment` |
| **Processing instruction** | `<?xml ...?>` | `ProcessingInstruction` |
| **Namespace** | Пространство имён | (в DOM отдельного типа нет — атрибут `xmlns`) |
| **Root (Document)** | Корень документа — виртуальный узел над `<html>` | `Document` |

**Чем модель XPath отличается от DOM на нюансах:**

| Аспект | DOM | XPath |
|---|---|---|
| Атрибуты | Узлы `Attr`, но **не** дети `Element` | Узлы оси `attribute::`, тоже не child |
| Соседние текстовые блоки | Могут быть разделены (`splitText()`) | Сливаются в один text-node |
| `CDATASection`, `EntityReference` | Отдельные типы | Не существуют — содержимое сливается с Text |
| `DocumentFragment` | Есть | Нет |

Для Selenium-тестов и обычного HTML эти отличия незаметны — в HTML нет CDATA и entity references, и работаем мы фактически с DOM. Различия всплывают только при применении XPath к произвольному XML.

**Свойства узла:** имя, значение, родитель, дети, атрибуты, namespace.

**Результат XPath-выражения:** **множество узлов** (node-set), которое при необходимости приводится к строке, числу или булеву (см. `string()`, `number()`, `boolean()` в [3.7](#37-системные-функции-xpath-10)).

## 3.3. Типы выражений

| Тип | Возвращает | Пример |
|---|---|---|
| Path expression | Node-set | `//a[@class='link']` |
| Boolean expression | true/false | `contains(@href,'/ask')` |
| Numeric expression | Число | `count(//a) + 1` |
| String expression | Строку | `concat(@id, '_', @name)` |

## 3.4. Абсолютные и относительные пути

```
/html/body/div/p          — абсолютный путь (от корня)
//div[@class='post']      — относительный (где угодно в дереве)
.                         — текущий узел
..                        — родительский узел
@href                     — атрибут href
*                         — любой элемент
text()                    — текстовый узел
node()                    — любой узел
```

| Конструкция | Значение |
|---|---|
| `/A/B` | Дочерний `B` элемента `A` (один уровень) |
| `/A//B` | Любой потомок `B` элемента `A` (любая глубина) |
| `//B` | Все `B` в документе |
| `.//B` | Все `B` среди потомков текущего узла |

В нашем проекте используются **только относительные** XPath с `//`, потому что они устойчивее к изменению верхнего уровня вёрстки:

```java
By.xpath("//h1[normalize-space()]")
By.xpath("//a[contains(@href,'/questions/ask') and contains(normalize-space(.),'Ask Question')]")
```

## 3.5. Предикаты

Предикат `[...]` фильтрует узлы. Предикатов может быть несколько подряд — применяются по очереди слева направо.

| Выражение | Что выбирает |
|---|---|
| `//a[1]` | Первый `<a>` (положение) |
| `//a[last()]` | Последний |
| `//a[position()=2]` | Второй |
| `//div[@class='post']` | По атрибуту |
| `//div[@class='post'][2]` | Второй из подходящих |
| `//div[count(p)>3]` | Где более 3 параграфов |
| `//input[@type='text' and @name='q']` | По нескольким условиям |
| `//p[@class='note' or @class='warn']` | По любому из условий |

Пример из нашего проекта — выбор первой ссылки на вопрос:
```xpath
(//div[contains(@class,'s-post-summary')]//h3//a[contains(@href,'/questions/')])[1]
```
Скобки вокруг пути нужны, чтобы `[1]` применился к **результату всего выражения**, а не к одной ссылке.

## 3.6. Оси (axes)

Ось определяет **направление навигации** от текущего узла. Полная форма шага: `axis::node-test[predicate]`.

| Ось | Что обходит |
|---|---|
| `child::` | Прямые дети (по умолчанию, можно опустить) |
| `descendant::` | Все потомки |
| `descendant-or-self::` | Сам узел + потомки (это и есть `//`) |
| `parent::` | Прямой родитель |
| `ancestor::` | Все предки |
| `ancestor-or-self::` | Сам узел + предки |
| `following::` | Все узлы документа после закрытия текущего |
| `following-sibling::` | Братья ниже |
| `preceding::` | Все узлы документа до открытия текущего |
| `preceding-sibling::` | Братья выше |
| `attribute::` | Атрибуты (сокращение `@`) |
| `self::` | Сам узел (сокращение `.`) |
| `namespace::` | Пространства имён |

### 3.6.1. Сокращения

| Сокращение | Полная форма |
|---|---|
| `//` | `/descendant-or-self::node()/` |
| `@attr` | `attribute::attr` |
| `.` | `self::node()` |
| `..` | `parent::node()` |
| `*` | `child::*` |

### 3.6.2. Примеры с осями

```xpath
//label[text()='Email']/following-sibling::input
   — input справа от label «Email»

//input[@id='q']/ancestor::form
   — форма, в которой лежит input

//td[normalize-space()='Active']/preceding-sibling::td[1]
   — соседняя ячейка слева от ячейки «Active»
```

В нашем проекте мы оси явно не используем (все запросы — это `//element[predicate]`), но знание осей критично для сложных страниц, особенно когда нужно «найти элемент по соседнему элементу».

## 3.7. Системные функции XPath 1.0

XPath 1.0 определяет фиксированный набор функций. Группируются по типу возвращаемого значения.

### 3.7.1. Функции над node-set

| Функция | Что делает |
|---|---|
| `position()` | Индекс текущего узла в текущей выборке (1-based) |
| `last()` | Размер текущей выборки |
| `count(node-set)` | Количество узлов |
| `name()` | Полное имя текущего узла |
| `local-name()` | Локальное имя (без префикса namespace) |
| `namespace-uri()` | URI пространства имён |
| `id('foo')` | Узел с `id="foo"` (требует DTD; в HTML обычно работает) |

### 3.7.2. Строковые функции

| Функция | Что делает |
|---|---|
| `string(obj)` | Преобразование к строке |
| `concat(s1, s2, ..., sN)` | Конкатенация |
| `starts-with(s, prefix)` | `s` начинается с `prefix`? |
| `contains(s, sub)` | `s` содержит `sub`? |
| `substring(s, start)` / `substring(s, start, len)` | Подстрока (1-based) |
| `substring-before(s, sub)` | Часть `s` до первого `sub` |
| `substring-after(s, sub)` | Часть `s` после первого `sub` |
| `string-length(s)` | Длина |
| `normalize-space(s)` | Убирает ведущие/концевые пробелы и схлопывает повторы |
| `translate(s, from, to)` | Поэлементная замена символов (используется как case-fold в 1.0) |

### 3.7.3. Числовые функции

| Функция | Что делает |
|---|---|
| `number(obj)` | Преобразование к числу |
| `sum(node-set)` | Сумма числовых значений узлов |
| `floor(x)`, `ceiling(x)`, `round(x)` | Округления |

### 3.7.4. Булевы функции

| Функция | Что делает |
|---|---|
| `boolean(obj)` | Преобразование к булеву (node-set непуст ⇒ true) |
| `not(b)` | Отрицание |
| `true()`, `false()` | Литералы |
| `lang(code)` | Узел в указанном языке (по `xml:lang` предков) |

### 3.7.5. Чего НЕТ в XPath 1.0

| Хочется | В XPath 1.0 нет | Обходной путь / появилось в 2.0 |
|---|---|---|
| `lower-case()` / `upper-case()` | Нет | `translate(s, 'ABC...', 'abc...')` |
| Регэкспы (`matches`, `replace`) | Нет | XPath 2.0 |
| `if/then/else` в выражении | Нет | XPath 2.0 |
| `for ... return` | Нет | XPath 2.0 |
| Функции даты-времени | Нет | XPath 2.0 |
| Группировка / сортировка | Нет | XSLT/2.0 |

Selenium застрял в 1.0 (это ограничение браузерного `document.evaluate`), поэтому идиома `translate` для case-insensitive поиска — повсеместная.

## 3.8. Работа с множествами элементов

XPath изначально оперирует **множествами узлов** (node-set), а не одним элементом. Это даёт мощные возможности.

### 3.8.1. Объединение `|`

Возвращает объединение двух node-set'ов:

```xpath
//h1 | //h2
   — все h1 и h2

//div[@class='question']//div[@class='s-prose'] | //div[contains(@class,'s-prose') and normalize-space()]
```
Второй пример — реальный локатор `QUESTION_BODY` из нашего проекта: либо `s-prose` внутри блока вопроса, либо любой непустой `s-prose` (запасной вариант).

### 3.8.2. Логические операторы внутри предиката

`and`, `or`, `not()` — комбинирование условий:

```xpath
//input[(contains(@name,'q') or contains(@aria-label,'Search')) and not(@type='hidden')]
```
— из локатора `MAIN_SEARCH`. Скобки нужны для явного приоритета.

### 3.8.3. Применение функций к множеству

```xpath
count(//div[@class='post'])           — сколько постов на странице
sum(//td[@class='price'])             — сумма цен (если они числовые)
//ul[count(li) >= 5]                  — списки с ≥5 пунктов
```

### 3.8.4. Позиционные предикаты

```xpath
//tr[position() <= 3]                 — первые 3 строки таблицы
//li[position() = last()]             — последний пункт
//tr[position() mod 2 = 0]            — чётные строки
```

### 3.8.5. Селекция через relationship с другими узлами

Сила XPath по сравнению с CSS-селекторами — навигация **в любом направлении**:

```xpath
//th[normalize-space()='Status']/../td[3]
   — третья td в той же строке таблицы, где th = «Status»

//label[text()='Password']/following-sibling::input[1]
   — input сразу после label «Password»

//a[contains(.,'Download')]/ancestor::div[@class='card'][1]
   — ближайший div.card вокруг ссылки «Download»
```

### 3.8.6. Поиск по тексту

```xpath
//button[normalize-space()='Submit']             — точное совпадение
//*[contains(text(), 'Welcome')]                 — содержит подстроку
//*[starts-with(normalize-space(), 'Step ')]     — начинается с «Step »
//*[normalize-space() = 'X']                     — все потомки текста схлопнуты
```

`text()` возвращает только **прямой** текстовый узел, а `.` — конкатенацию всех текстовых потомков. Поэтому для надёжности лучше:

```xpath
//*[normalize-space(.)='Active']
```

### 3.8.7. Реальные примеры из нашего проекта

```xpath
//form[contains(@action,'/search')]
   //input[(contains(@name,'q') or contains(@aria-label,'Search'))
           and not(@type='hidden')]
```
Делает три вещи:
1. Находит форму, чей `action` содержит `/search`
2. Внутри неё ищет input
3. Принимает либо `name~='q'`, либо `aria-label~='Search'`, отбрасывая скрытые поля

```xpath
//div[contains(@class,'s-post-summary') or contains(@class,'question-summary')]
```
Поддержка двух поколений вёрстки Stack Overflow одним выражением.

```xpath
//button[contains(translate(normalize-space(.),
                            'ABCDEFGHIJKLMNOPQRSTUVWXYZ',
                            'abcdefghijklmnopqrstuvwxyz'),
                  'accept')
         or contains(translate(...), 'necessary')]
```
Шаг за шагом:
1. `normalize-space(.)` — собирает весь текст кнопки и схлопывает пробелы
2. `translate(..., 'ABC...', 'abc...')` — приводит к нижнему регистру (случай 1.0-замены `lower-case()`)
3. `contains(..., 'accept')` — ищет «accept» в нижнем регистре

Так одна XPath-конструкция ловит «Accept all», «I accept», «accept cookies», «Only Necessary» — все варианты cookie-баннеров Stack Overflow.

## 3.9. Производительность XPath

| Конструкция | Стоимость |
|---|---|
| `//element` | Дорого — обходит весь документ |
| `/абсолютный/путь` | Дёшево — один проход по детям |
| `descendant::` (`//` внутри) | Дорого |
| Сложные предикаты с `position()` | Средне |
| `//element[@id='x']` | Дёшево, если есть индекс по id |

Практическое правило: для больших страниц предпочитайте контекстные запросы вместо `//`:

```xpath
не: //form//input[@name='q']
а:  //form[contains(@action,'/search')]/descendant::input[@name='q']
```

Для тестов это редко критично — UI-тесты медленные сами по себе из-за сетевых задержек и рендеринга.

## 3.10. Сравнение XPath и CSS

| Характеристика | CSS | XPath |
|---|---|---|
| Скорость в браузерах | Чуть быстрее | Чуть медленнее |
| Навигация вверх (parent, ancestor) | Нет | Да |
| Поиск по тексту | Нет (только `:contains` в jQuery, не CSS) | `contains()`, `text()=...` |
| Сложные предикаты (`and`/`or`/функции) | Ограниченно | Полноценно |
| Читаемость простых селекторов | Лучше | Хуже |
| Поддержка во всех инструментах | Везде | Везде |
| Версионирование/стандарт | CSS Selectors L4 | XPath 1.0/2.0/3.0 |

Эмпирическое правило: простые случаи (по классу, ID, тегу, дочерним отношениям) — CSS. Сложные (по тексту, по соседним элементам, с условной логикой) — XPath.

В нашей лабе использование XPath обусловлено требованием задания: «выбор элементов DOM не основан на ID». XPath даёт максимальную выразительность для семантического поиска без ID.
